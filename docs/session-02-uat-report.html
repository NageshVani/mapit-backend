● Critical
BUG-01
verifyOtp() missing needsProfile check — OTP users bypass profile setup
✓ Fixed
Where
📄 MapIt_MVP_v1.html · verifyOtp() function
Root Cause
submitAuth() (the email+password path) was updated in commit 3d100ab with a needsProfile check that catches old accounts where nickname === 'MapIt User' or full_name === 'MapIt User', routing them to step-profile even when isNewUser = false (profile row already exists with placeholder data).

However, verifyOtp() — the OTP fallback path — only checked data.isNewUser. Any existing user with a placeholder nickname who clicks "Use email OTP →" and verifies their code would be taken directly into the app with "MapIt User" as their display name, bypassing profile setup entirely.
Impact
All family members who originally registered via OTP (before Session 2) and attempt to use the OTP fallback would never be prompted to set their real nickname. They would appear in "Who's using MapIt?" as "MapIt User" regardless of the DB fixes applied.
Fix Applied
MapIt_MVP_v1.html — verifyOtp()
    // after verifying OTP and fetching profile...
-   if(data.isNewUser){
-     goToStep('step-profile');
-   } else {
+   const needsProfile = data.isNewUser
+     || !data.profile?.nickname
+     || data.profile.nickname === 'MapIt User'
+     || !data.profile?.full_name
+     || data.profile.full_name === 'MapIt User';
+   if(needsProfile){
+     if(data.profile?.full_name && data.profile.full_name !== 'MapIt User')
+       document.getElementById('nameInput').value = data.profile.full_name;

● Medium
BUG-02
_refreshStaleSessionNames() skips sessions with nickname = 'MapIt User'
✓ Fixed
Where
📄 MapIt_MVP_v1.html · _refreshStaleSessionNames()
Root Cause
On init(), this function background-fetches /api/auth/me for every localStorage session that lacks a proper display name, updating the cache so "Who's using MapIt?" shows correct names without requiring a click-through.

The filter condition was !u.nickname — which is false (truthy) when u.nickname === 'MapIt User'. Therefore, sessions where localStorage cached the placeholder string were silently excluded from the refresh, meaning stale "MapIt User" buttons would persist on the Who's using MapIt? screen even after the DB was corrected.
Fix Applied
MapIt_MVP_v1.html — _refreshStaleSessionNames()
  const stale = Object.entries(sessions).filter(([, u]) =>
-   u.session?.access_token && !u.nickname
+   u.session?.access_token && (!u.nickname || u.nickname === 'MapIt User')
  );
● Medium
BUG-03
submitAuth() Sign-In path doesn't pre-fill profileNicknameInput from existing profile
✓ Fixed
Where
📄 MapIt_MVP_v1.html · submitAuth()
Root Cause
When needsProfile is true and the user came via the Sign In tab (not Create Account), ST._pendingNickname is null because no nickname was captured in the sign-in form. The existing code pre-filled nameInput (full name) from the profile fallback, but profileNicknameInput was only pre-filled from ST._pendingNickname — leaving the nickname field blank.

A user signing in and being sent to profile setup would need to manually type a nickname even if their profile already had a valid one (just not as the primary display method).
Fix Applied
MapIt_MVP_v1.html — submitAuth() → step-profile pre-fill
      if(ST._pendingNickname) document.getElementById('profileNicknameInput').value = ST._pendingNickname;
+     else if(data.profile?.nickname && data.profile.nickname !== 'MapIt User')
+       document.getElementById('profileNicknameInput').value = data.profile.nickname;
      goToStep('step-profile');

⚠ Observations (Non-Blocking)
These are not bugs — nothing is broken. They are noted for awareness and future action where appropriate.
# Type Observation Recommended Action
O1 Config OTP_RATE_LIMIT=100 in local .env — Express OTP limiter set to 100/hr instead of intended 10/hr. Supabase's own 60s cooldown provides adequate protection locally. Verify Vercel env var is set to 10 in both UAT and production projects.
O2 Info OTP rate limiting is double-layered: Supabase imposes 60s per-email cooldown (observed in tests); Express adds a per-IP hourly limit. Together they provide adequate protection. No single point of failure. No action needed. Both layers working as intended.
O3 Security resetSentMsg in sendPasswordReset() uses innerHTML with the raw email string. A crafted email like <img src=x onerror=alert(1)>@test.com would pass the frontend @ check but fail Supabase validation. Low risk but textContent is always safer. Optional hardening before public launch: escape email before inserting into innerHTML.
O4 UX AUTH_MODE variable is not reset to 'signin' when navigating back to step-auth from setNewPassword() or continueAs() session expiry. On a fresh page load this is always 'signin'; only a rare interactive edge-case could leave it in register mode. Acceptable for MVP. Can call switchAuthTab('signin') in those paths if needed before public launch.
O5 UX Password fields in step-auth are not cleared when switching between Sign In and Create Account tabs. Previous input persists. Unlikely to cause confusion for typical use patterns. Acceptable for MVP.
