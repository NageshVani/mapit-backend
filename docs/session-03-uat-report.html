generateRefCode() implementation

function generateRefCode() {
  const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'; // no 0/O/1/I
  let code = 'MP-BLR-';
  for (let i = 0; i < 6; i++) code += chars[Math.floor(Math.random() * chars.length)];
  return code;
}
// e.g. MP-BLR-A3K9X2 — 32^6 = ~1 billion combinations

Observations

OBS-01
Reference code alphabet designed to prevent visual confusion
ABCDEFGHJKLMNPQRSTUVWXYZ23456789 — excludes 0, O, 1, I which look similar in many fonts/handwriting. When a buyer calls a seller and says "MP-BLR-A3K9X2", there's no ambiguity. 32^6 = ~1.07 billion combinations — effectively collision-free for MVP scale.
OBS-02
views_count bug fix: grep false positive explained
The automated test B7 initially reported 2 stale view_count references. Investigation revealed both were review_count (a seller reputation column on the profiles table) — the grep matched as a substring. Zero actual listings.view_count references remain. Bug is fully fixed.
OBS-03
Existing listings have NULL reference_code and expires_at
Migration 002 adds columns but does not backfill existing rows. All listings created before Session 3 will have reference_code = NULL and expires_at = NULL. The frontend handles this gracefully — the Ref line is conditionally rendered (${l.reference_code ? … : ''}) and expiry countdown only shows when expires_at is set. No UI breakage for old listings.
OBS-04
show_phone preference not yet visible to buyers in detail view
The show_phone field is stored and returned by the API but the buyer-facing listing detail view does not yet act on it (i.e., hide/show the seller's phone number conditionally). This is intentional — the seller's phone is not displayed anywhere in the current UI anyway. When the contact flow is built (Session 5/6), show_phone will gate whether the phone number is shown to buyers. The field is ready and stored.
OBS-05
Auto-expiry enforcement not yet built (cron deferred to Session 7)
expires_at is stored correctly but no server-side process currently changes listing status to 'expired' when the date passes. Expired listings will continue to appear as 'active' until manually marked. The expiry countdown in My Ads is informational only for now. Automated expiry via Vercel Cron or Supabase Edge Function is planned for Session 7.
OBS-06
Delete removes listing but not its photos from Supabase Storage
DELETE /api/listings/:id deletes the DB row but does not call Supabase Storage to remove the associated photo files. For MVP family-beta scale this is acceptable (storage quota is ample). Before public launch, add a storage cleanup step in the DELETE route: fetch listing_photos rows first, delete storage objects, then delete the DB row.
OBS-07 · 2026-06-27
openMyAds() tab-active bug fixed as a side-effect
The original openMyAds() removed 'on' from browse and fav tab buttons but never added 'on' to the "My Ads" tab button — so opening My Ads from the profile menu left the tab bar with no highlighted tab. Refactoring to switchTab('mine') fixed this as a zero-cost side-effect of the proximity-radius work.
OBS-08 · 2026-06-27
listingRow keepTab evaluated at click time — future-proof
The listingRow() onclick attribute uses ST.tab==='fav'||ST.tab==='mine' as a runtime expression, not a value captured at render time. This means if the tab ever changes between render and click (unlikely but possible during async re-renders), the correct behaviour is always applied. This pattern is preferable to passing a boolean at render time.
OBS-09 · 2026-06-27
Saved tab map is independent of proximity radius — correct design
Saved listings are user-curated — they have personal relevance regardless of where the user is looking on the map. Tying them to the proximity radius would hide saved listings in other parts of the city. The separate ST.favListings[] array and disabled radius confirm this architectural boundary. The map auto-fits to saved pins so the user always sees all of them.
