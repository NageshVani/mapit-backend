⚠ Observations & Non-Blocking Notes
No code bugs were found in Session 1 deliverables. The following observations surfaced during the UAT — they are either resolved constraints, known limitations, or items to watch. None block merging uat → main.
Config uat.mapit.co.in always points to production (main), not the uat branch OBS-01 ⊙ Known — Monitor
What Was Found

Vercel Free Tier limitation: branch-specific custom domains require a Pro plan. uat.mapit.co.in is registered in the Vercel project but points to the production deployment (main branch), not the uat branch preview. This is confirmed by the fact that both uat.mapit.co.in and www.mapit.co.in return identical responses.

Impact & Mitigation

UAT testing must use the Vercel preview URL: https://mapit-backend-git-uat-nagesh-n-arun.vercel.app. This was confirmed working (200 OK). All UAT checklist links use the preview URL, not uat.mapit.co.in. If Vercel Pro is upgraded in the future, branch-specific domains can be configured.

Data Migration 001 columns were missing from Supabase until 2026-06-21 re-apply OBS-02 ✓ Fixed
What Was Found

CONTEXT.md originally stated Migration 001 was applied, but a manual inspection on 2026-06-21 confirmed that 5 columns were missing from the profiles table: nickname, home_lat, home_lng, home_address, agreed_tos_at. auth_provider and default_view had been applied correctly.

Resolution

All 5 missing columns were added via a single BEGIN/COMMIT block in the Supabase SQL Editor on 2026-06-21. The migration script uses IF NOT EXISTS on every ALTER TABLE, so re-running was safe — it silently skipped the 2 columns that already existed and only added the 5 missing ones. Nickname was then backfilled: UPDATE profiles SET nickname = full_name WHERE nickname IS NULL.

Infrastructure Single Supabase project serves both UAT and production OBS-03 ⊙ Accepted — Revisit Pre-Public
What Was Found

There is only one Supabase project (map it-backend). UAT tests and production traffic both read from and write to the same database. Any schema change or data inserted during UAT is immediately visible in production.

Rationale

Accepted for the family-beta MVP phase (12 users, no paying customers). The overhead of maintaining two database schemas is not justified at this stage. Decision to revisit before public launch or before any destructive schema migration is needed (e.g., a column type change that would require a data migration).

Security OTP_RATE_LIMIT is set to 100 in local .env — Vercel production value must be verified OBS-04 ⊙ Verify Manually
What Was Found

The local .env file has OTP_RATE_LIMIT=100 (per hour per IP). This is intentionally high for local development to avoid hitting the limit during test cycles. The Vercel production environment variable should be set to 10 (10 OTP sends per IP per hour) to match the intended security posture.

Action Required

Verify in Vercel Dashboard → Settings → Environment Variables → OTP_RATE_LIMIT. Supabase's built-in 60-second OTP cooldown provides a secondary guard even if the Express rate limit is misconfigured. However, the Vercel value should explicitly be 10 to avoid a large number of OTP sends from a single abusive IP.

Data invite_codes.used_by is NULL for all rows — backfill not possible without email column OBS-05 Deferred — Invite codes archived
What Was Found

All rows in the invite_codes table have used_by = NULL. This means the profiles.full_name auto-backfill SQL (UPDATE profiles SET full_name = ic.created_for FROM invite_codes ic WHERE profiles.id = ic.used_by) would affect zero rows even if run. The invite_codes table also has no email column, ruling out a join via auth.users.email.

Impact & Resolution

Family member names were set manually by Nagesh in the Supabase table editor after identifying UUIDs via a SELECT p.id, u.email FROM profiles JOIN auth.users ON u.id = p.id join query. The needsProfile check added in Session 2 now prevents placeholder names ('MapIt User') from persisting in new accounts. This issue is fully mitigated going forward. Invite codes are archived — the table is kept for historical record but is no longer used in auth.

Data Some invite_codes.created_for names have "1" appended (e.g. "Chirag 1") OBS-06 ⊙ Fix Before Login
What Was Found

Four invite_codes rows have created_for values with a trailing "1": Chirag 1, Kalpith 1, Srikanth 1, Vijay 1. This appears to be an accidental duplicate entry from the Supabase table editor where rows were inserted twice and the second row got " 1" appended.

Action Required

Fix in Supabase Table Editor before the corresponding family members log in. Run: UPDATE invite_codes SET created_for = trim(trailing ' 1' from created_for) WHERE created_for LIKE '% 1'; — or edit manually in the table editor. Since invite codes are now archived (not used for auth), this only affects historical display; it does not block any functionality.

💡 Key Architectural Decisions Made in Session 1
Decisions recorded here because they constrain future sessions. Each was a deliberate trade-off, not a default.
Decision Rationale & Implication
Frontend served from Express backend (not a separate Vercel project) Vercel always reads the root vercel.json regardless of "Root Directory" setting — routing everything to Express. A separate static Vercel project would require a separate repo. Single-project approach is simpler and avoids CORS complexity. Trade-off: one cold-start handles both UI and API requests.
Helmet CSP disabled globally (contentSecurityPolicy: false) The single-file frontend uses inline <script> blocks and loads Leaflet, Font Awesome, and Google Fonts from CDNs. Default Helmet CSP blocks all of these. Disabling CSP entirely is acceptable for MVP but should be replaced with a proper CSP nonce strategy before public launch.
Two-file rule: MapIt_MVP_v1.html ↔ public/index.html must always be byte-identical The canonical frontend is edited in the root as MapIt_MVP_v1.html (for readability and git history), then synced to public/index.html (the deployed copy). The diff check is the commit gate. This rule must be enforced on every frontend change.
API_URL auto-detects via window.location.hostname IIFE Eliminates the error-prone manual comment/uncomment cycle before every commit. localhost → port 3001 (local Express), mapit.co.in → api.mapit.co.in, any other host → same-origin (UAT/Vercel preview). Zero config across all environments.
Migrations pre-written in database/migrations/ before their session Avoids time spent writing SQL at the start of each session. The session can begin with applying the migration and immediately building features. All migrations use IF NOT EXISTS so they are safe to re-run.
dev / uat / main branch strategy dev = local-only work; uat = Vercel preview, family sign-off; main = production auto-deploy. No direct commits to main. Feature branches named feat/session-NN-description. Conventional commits enforced (feat/fix/docs prefix + scope).
